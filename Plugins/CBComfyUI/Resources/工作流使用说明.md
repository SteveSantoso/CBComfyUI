# CBComfyUI 工作流使用说明

这份说明对应 [工作流.json](工作流.json)。

## 1. 这份工作流在做什么

这不是 ComfyUI 画布布局用的前端 workflow，而是可以直接发给 ComfyUI `/prompt` 接口的 prompt JSON。它和插件的 `QueueWorkflowFromFile` / `QueueWorkflowFromString` 是直接兼容的。

这条链路的核心逻辑是：

- UE 插件先上传 `depth.png`、`normals.png`、`color.png`、`edge_mask.png`
- 当前工作流只使用 `depth.png`
- ComfyUI 用 `FLUX.1-dev + Union ControlNet(depth) + Nunchaku 4bit DiT` 生成结果图
- `SaveImage` 节点输出最终图片
- UE 再把输出图投回到目标 Actor 的材质上

从节点上看，关键依赖如下：

- 节点 `19`：`LoadImage`，固定读取 `depth.png`
- 节点 `23`：`ControlNetLoader`，读取深度 ControlNet
- 节点 `26`：`NunchakuFluxDiTLoader`，读取 4bit FLUX DiT 主模型
- 节点 `16`：`DualCLIPLoader`，读取 `t5xxl_fp16.safetensors` 和 `clip_l.safetensors`
- 节点 `17`：`VAELoader`，读取 `flux-ae.safetensors`
- 节点 `3`：`SaveImage`，返回最终输出文件名

## 2. 先决条件

在 ComfyUI 侧，至少要满足这些条件：

- 使用较新的 ComfyUI，确保内置 FLUX 相关节点可用
- 安装 `ComfyUI-nunchaku`，因为工作流里用了 `NunchakuFluxDiTLoader`
- 安装 Nunchaku 后端 wheel
- 使用 NVIDIA GPU；Nunchaku 是面向低比特推理的插件

Nunchaku 官方文档：

- 安装：https://nunchaku.tech/docs/ComfyUI-nunchaku/get_started/installation.html
- 使用：https://nunchaku.tech/docs/ComfyUI-nunchaku/get_started/usage.html

## 3. 模型清单

这份工作流至少需要下面这些模型文件。文件名最好和工作流里保持一致；如果你下载到的文件名不同，就改工作流节点里的名字。

| 节点 | 工作流中的文件名 | 作用 | 建议放置目录 |
| --- | --- | --- | --- |
| `16` | `t5xxl_fp16.safetensors` | FLUX 文本编码器 T5 | `ComfyUI/models/text_encoders` |
| `16` | `clip_l.safetensors` | FLUX 文本编码器 CLIP-L | `ComfyUI/models/text_encoders` |
| `17` | `flux-ae.safetensors` | FLUX VAE | `ComfyUI/models/vae` |
| `23` | `Shakker-LabsFLUX.1-dev-ControlNet-Union-Pro-2.0.safetensors` | FLUX depth ControlNet | `ComfyUI/models/controlnet` |
| `26` | `svdq-int4_r32-flux.1-dev.safetensors` | Nunchaku 4bit FLUX DiT 主模型 | `ComfyUI/models/unet` |

额外说明：

- `flux-ae.safetensors` 有些仓库里文件名叫 `ae.safetensors`。你可以直接改节点 `17` 的 `vae_name`，也可以把文件重命名成 `flux-ae.safetensors`。
- `FLUX.1-dev` 相关权重通常需要先在 Hugging Face 接受许可。
- `Shakker-Labs/FLUX.1-dev-ControlNet-Union-Pro-2.0` 也沿用了 FLUX.1-dev 的许可体系。
- Nunchaku 官方文档对 Blackwell / RTX 50 系显卡建议优先使用 FP4 模型；你这份工作流当前写死的是 INT4 文件名，如果你换成 FP4，需要同步修改节点 `26` 的 `model_path`。

## 4. 推荐下载方式

### 4.1 文本编码器

官方 Nunchaku 文档给出的目录就是 `models/text_encoders`：

```bash
hf download comfyanonymous/flux_text_encoders clip_l.safetensors --local-dir models/text_encoders
hf download comfyanonymous/flux_text_encoders t5xxl_fp16.safetensors --local-dir models/text_encoders
```

### 4.2 VAE

可以从 Black Forest Labs 的 FLUX 仓库下载 VAE：

```bash
hf download black-forest-labs/FLUX.1-dev ae.safetensors --local-dir models/vae
```

如果你不改工作流，就把 `ae.safetensors` 重命名为 `flux-ae.safetensors`。

### 4.3 Nunchaku 4bit FLUX 主模型

Nunchaku 官方文档给出的示例目录是 `models/unet`：

```bash
hf download nunchaku-tech/nunchaku-flux.1-dev svdq-int4_r32-flux.1-dev.safetensors --local-dir models/unet
```

如果 `nunchaku-tech` 仓库名后续调整，按 Nunchaku 官方文档或官方 Hugging Face 组织当前页面为准。

### 4.4 Shakker Union ControlNet

从下面这个仓库的 Files 页面下载与节点 `23` 对应的 safetensors 文件：

- https://huggingface.co/Shakker-Labs/FLUX.1-dev-ControlNet-Union-Pro-2.0

放到：

- `ComfyUI/models/controlnet`

如果实际文件名不是 `Shakker-LabsFLUX.1-dev-ControlNet-Union-Pro-2.0.safetensors`，就把节点 `23` 的 `control_net_name` 改成实际文件名。

## 5. 这条工作流和插件怎么对接

插件的相关入口在这些位置：

- [../Source/CBComfyUI/Public/CBComfyUIManager.h](../Source/CBComfyUI/Public/CBComfyUIManager.h)
- [../Source/CBComfyUI/Public/CBComfyUISceneCaptureComponent.h](../Source/CBComfyUI/Public/CBComfyUISceneCaptureComponent.h)
- [../Source/CBComfyUI/Private/CBComfyUISceneCaptureComponent.cpp](../Source/CBComfyUI/Private/CBComfyUISceneCaptureComponent.cpp)

当前插件实际会上传这四张控制图：

- `depth.png`
- `normals.png`
- `color.png`
- `edge_mask.png`

这一点在 [../Source/CBComfyUI/Private/CBComfyUISceneCaptureComponent.cpp#L129](../Source/CBComfyUI/Private/CBComfyUISceneCaptureComponent.cpp#L129) 附近可以看到。

工作流提交入口和节点参数修改入口分别在：

- [../Source/CBComfyUI/Public/CBComfyUIManager.h#L152](../Source/CBComfyUI/Public/CBComfyUIManager.h#L152)
- [../Source/CBComfyUI/Public/CBComfyUIManager.h#L222](../Source/CBComfyUI/Public/CBComfyUIManager.h#L222)

`QueueWorkflowFromFile` 的原始文件读取实现见：

- [../Source/CBComfyUI/Private/CBComfyUIManager.cpp#L211](../Source/CBComfyUI/Private/CBComfyUIManager.cpp#L211)

但你的工作流里只有节点 `19` 在读取 `depth.png`，所以当前版本实际上只吃深度图。

### 5.1 UE 侧最小调用顺序

推荐顺序如下：

1. 在关卡里放一个 `CBComfyUIManager`
2. 把 `ComfyURL` 设成你的 ComfyUI 地址，默认是 `http://127.0.0.1:8188`
3. 运行时先调用 `ConnectCommfyUI`
4. 打包环境建议先调用 `ConfigureForRuntime`
5. 给 `CBComfyUISceneCaptureComponent` 设置：
   - `CaptureCamera`
   - `ComfyUIManager`
   - `BaseMaterial`
   - `ReferenceTexture`
6. 调用 `PrepareActorsRuntime` 或 `PrepareActorsAuto`
7. 调用 `ProcessMultipleActors` 上传控制图
8. 调用 `QueueWorkflowFromFile` 或 `QueueWorkflowFromString` 提交这份 JSON
9. 监听 `OnComfyUIOutputFile`
10. 取返回文件名数组里的第一张图，传给 `ProcessRenderResults`

### 5.2 蓝图里最关键的两个事件

一个是提交工作流：

- `CBComfyUIManager -> QueueWorkflowFromFile`

一个是接收结果图：

- `CBComfyUIManager -> OnComfyUIOutputFile`

拿到输出文件名后，再把文件名传给：

- `CBComfyUISceneCaptureComponent -> ProcessRenderResults`

插件内部就是用这个文件名再去下载 ComfyUI 输出图，然后回写到 Actor 纹理上。

## 6. 如果你想在 UE 里动态改 prompt 或参数

这份 JSON 的节点 ID 就是字符串形式的数字，所以你在 UE 里改参数时，节点名直接传这些数字字符串即可。

常用可改项：

- 节点 `1` 的 `text`：正向 prompt
- 节点 `18` 的 `text`：负向 prompt
- 节点 `5` 的 `steps`：采样步数
- 节点 `8` 的 `guidance`：FLUX guidance
- 节点 `25` 的 `strength`：ControlNet 强度
- 节点 `21` 的 `sampler_name`：采样器名

对应接口在 [../Source/CBComfyUI/Public/CBComfyUIManager.h#L152](../Source/CBComfyUI/Public/CBComfyUIManager.h#L152) 和 [../Source/CBComfyUI/Public/CBComfyUIManager.h#L222](../Source/CBComfyUI/Public/CBComfyUIManager.h#L222)。

一个典型做法是：

1. 先把 JSON 读成 `UCBJsonObject`
2. 调 `SetNodeInputStringProperty` / `SetNodeInputIntProperty` / `SetNodeInputDoubleProperty`
3. 再调 `QueueWorkflowFromJsonObject`

## 7. 打包版要额外注意的事

### 7.1 Static Mesh 必须开启 Allow CPU Access

插件在运行时会读取静态网格的 CPU 顶点、UV、索引数据，再把生成图投回模型表面。

所以所有要参与这条流程的 Static Mesh，都要在资源里开启：

- `Allow CPU Access`

否则 packaged build 里这些 CPU 数据会被 cook 裁掉，结果就是回写阶段失败。

### 7.2 工作流 JSON 是原始文件，不是 UAsset

`QueueWorkflowFromFile` 走的是原始文件路径，不是资源引用。实现见：

- [../Source/CBComfyUI/Private/CBComfyUIManager.cpp#L211](../Source/CBComfyUI/Private/CBComfyUIManager.cpp#L211)

这意味着打包版里不要默认指望 `Plugins/CBComfyUI/Resources/工作流.json` 一定能被原样找到。更稳的做法有两个：

- 做法 A：在打包设置里把工作流 JSON 所在目录加入 `Additional Non-Asset Directories to Copy`
- 做法 B：运行时先把 JSON 读成字符串，再用 `QueueWorkflowFromString`

如果你是专门给打包版用，我更建议用做法 B。

## 8. 当前工作流的限制

这份工作流目前只使用深度图，不使用插件已经上传的：

- `normals.png`
- `color.png`
- `edge_mask.png`

如果你后面想增强约束，可以：

- 把节点 `19` 改成读取 `normals.png`
- 或者增加新的 `LoadImage` 节点，把 `edge_mask.png`、`color.png` 也接入
- 或者改成多条件 ControlNet

## 9. 建议的排查顺序

如果接上 UE 后没有出图，先按这个顺序查：

1. 先在 ComfyUI 本地手动确认模型都能被节点识别
2. 确认节点 `19` 读取到的确实是 `depth.png`
3. 确认 `CBComfyUIManager` 已连接成功
4. 确认 `OnComfyUIOutputFile` 能收到 `SaveImage` 返回的文件名
5. 确认目标 Static Mesh 开启了 `Allow CPU Access`
6. 确认基础材质里有名为 `BaseColor` 的纹理参数

如果你后面要把这套流程做成完全运行时可配置版本，建议再补一个专门的 Blueprint 示例：上传控制图、改 prompt、提交 workflow、收结果图、回写材质，全部串成一个明确的状态流。
